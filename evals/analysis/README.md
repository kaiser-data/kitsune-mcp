# analysis/

Post-hoc reports built from `results/`. Not auto-generated by this framework — your job to write the notebooks/scripts. The contract is just "read `results/**/score.json`, produce a report."

## Suggested artifacts

### `leaderboard.md.j2`
A Jinja2 template that produces a markdown table:

| System | Model | Category | pass@1 | pass@5 | σ | $/correct | wall p50 (s) |
|---|---|---|---|---|---|---|---|

Auto-rebuild on each batch via `make analysis`.

### `cost_vs_quality.ipynb`
Scatter plot:
- X: $ per task (mean across runs)
- Y: pass rate
- Color: system family
- Annotation: model

Useful for spotting Pareto-frontier systems.

### `variance_report.ipynb`
For each (system, task):
- Mean ± 95% CI
- Worst-decile score
- Bootstrap distribution of pass rate

Use this to decide whether differences between systems are statistically meaningful.

### `regression_alerts.md`
Diff today's batch vs. rolling-30-day average for the same (system, task) pair. Flag any drop > 2σ.

Provider-side drift is real — same model, same task, score changes month-to-month. Detecting it early matters.

### `category_radar.ipynb`
Spider plot, one axis per category, one polygon per system. Shows capability profile at a glance.

## Don't auto-generate, do version-control

The Pareto/variance plots are decision-grade artifacts. Commit them (rendered PDFs, or static HTML) so historical comparisons are reviewable. Don't trust "just re-run the notebook" — six months from now, the libraries have moved.
